# Contributing to SummerCamp-2026

Thank you for your interest in contributing to SummerCamp-2026!

This repository welcomes improvements to course materials, documentation, examples, and project showcases. Contributions from first-time open-source contributors are especially welcome.

## Before You Start

Before working on a change:

1. Check the existing Issues and Pull Requests to avoid duplicate work.
2. Open an Issue describing the problem or proposed improvement.
3. Wait for a maintainer to confirm that the proposed change fits the repository.

Small corrections, such as fixing an obvious typo, may be submitted directly. For larger documentation changes, please discuss the proposal in an Issue first.

## Contribution Workflow

### 1. Fork the repository

Create a personal copy of the repository by selecting **Fork** on GitHub.

### 2. Create a branch

Create a focused branch for your change. Use a short and descriptive branch name, for example:

```bash
git checkout -b docs/add-contributing-guide
```

Avoid making changes directly on the `main` branch of your fork.

### 3. Make your changes

Keep each contribution focused on one clearly defined issue.

For documentation changes:

* Use clear and concise language.
* Follow the existing Markdown style.
* Check headings, lists, links, and code blocks.
* Preview the rendered Markdown before submitting.

### 4. Commit your changes

Use a concise commit message that explains the purpose of the change, for example:

```bash
git commit -m "docs: add contribution guide"
```

### 5. Submit a pull request

Push the branch to your fork and open a Pull Request against the upstream repository's `main` branch.

In the Pull Request description:

* Explain what was changed.
* Explain why the change is useful.
* Link the related Issue.
* Keep unrelated changes out of the Pull Request.

### 6. Participate in review

Maintainers may ask questions or request revisions. Continue updating the same branch so that the Pull Request is updated automatically.

A contribution is complete after it has been reviewed and merged.

## Pull Request Checklist

Before submitting a Pull Request, confirm that:

* [ ] The change addresses one clear issue.
* [ ] The content has been proofread.
* [ ] Markdown renders correctly.
* [ ] Links and examples have been checked.
* [ ] No unrelated files were changed.
* [ ] The related Issue is referenced.

Thank you for helping improve SummerCamp-2026!
